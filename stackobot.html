Project Overview

Stack O Bot challenged me to design and implement two original gameplay mechanics within Unreal Engine's existing framework. I chose to create a defensive shield system with UI integration and a hazardous environmental area that interacted dynamically with the shield mechanic, creating layered resource management gameplay.

The first mechanic introduced a defensive shield pickup that activated a shield resource bar in the player UI, along with refillable shield pickups scattered throughout the level. The second mechanic featured a damageable traversal area where players with active shields would deplete shield resources before taking health damage, encouraging strategic resource management. Additionally, I implemented variable movement speed controlled through mouse wheel inputs, allowing players to dynamically adjust their traversal speed between slow precision movement and faster navigation.

As a solo developer working within time constraints, I was responsible for all design decisions, Blueprint implementation, UI integration, and iterative balancing. This project emphasized rapid prototyping and scope management while maintaining cohesive mechanical interactions.

Challenges & Solutions

Challenge 1: Shield Visual Attachment to Character Skeleton

The Problem: Attaching the shield visual element to the correct character bone within Unreal Engine's skeletal mesh system proved technically challenging. Initial attempts using simple component attachment resulted in the shield floating incorrectly relative to the character, not following animations properly, and displaying visual clipping through the character mesh during movement. The skeletal hierarchy contained dozens of bones, and identifying the correct attachment point required understanding bone naming conventions and skeletal structure. Additionally, the shield needed to maintain proper orientation and scale regardless of character animation state.

The Solution: Researched Unreal Engine's skeletal mesh component attachment system and implemented socket-based attachment using Blueprint's "Attach Component to Component" node with specific attachment rules. I navigated the character's skeletal hierarchy through the Skeleton Editor to identify the optimal bone (spine or chest bone) that provided stable positioning across all animation states. Created a custom socket on the chosen bone with precise location, rotation, and scale offsets to position the shield naturally. Used "Snap to Target" attachment rules with "Keep World" transform options to ensure the shield maintained correct positioning during character movement and animation blending. Tested attachment across various animation states (idle, walking, running, jumping) to verify consistent visual placement and adjusted socket transform values iteratively until achieving natural-looking shield positioning that moved organically with character motion.

Challenge 2: Dual-Damage System Blueprint Logic

The Problem: Implementing correct Blueprint logic for the dual-damage system where environmental hazards prioritize shield damage before health damage required complex conditional branching and state management. The system needed to handle multiple edge cases: what happens when shield is partially depleted and incoming damage exceeds remaining shield capacity? How to ensure damage overflow properly transfers to health without double-applying or losing damage values? How to maintain accurate UI updates reflecting both shield and health changes in real-time? Initial implementations either failed to transfer overflow damage correctly, updated UI inconsistently, or created situations where players could take health damage while shields remained active.

The Solution: Designed a comprehensive Blueprint function that handled all damage calculation logic in a single, testable system. Implemented branching logic using "Branch" nodes to first check shield status: if shield exists and is greater than zero, calculate remaining shield after damage, then use another branch to determine if damage exceeded shield capacity. If damage overflow occurred, calculated exact overflow amount (incoming damage minus remaining shield value), set shield to zero, then applied overflow damage to health. Used local variables within the function to store intermediate calculation values (remaining shield, overflow damage) preventing data loss between execution steps. Integrated "Set" nodes to update both shield and health values atomically, ensuring UI widgets bound to these variables reflected changes immediately. Added Blueprint validation using "Print String" debugging to verify damage calculations at each step during development, confirming accurate damage distribution across both resource types. Tested extensively with varying damage values, shield capacities, and health amounts to ensure robust behavior across all possible gameplay scenarios.

Challenge 3: Project Scope Management Under Time Constraints

The Problem: Managing project scope within tight time constraints while maintaining the original vision for complex interconnected mechanics became a critical challenge early in development. Initial design documentation outlined five ambitious mechanics including the shield system, multiple environmental hazard types, power-up variations, enemy AI interactions, and complex level progression systems. Midway through the project timeline, it became evident that implementing all planned features to a polished, bug-free state was unrealistic given available time. Continuing with the original scope risked delivering multiple half-finished, poorly-integrated systems rather than cohesive, fully-functional mechanics. Additionally, the interconnected nature of planned features meant incomplete systems would negatively impact the functionality of dependent mechanics.

The Solution: Conducted honest assessment of development progress and remaining time, then prioritized features based on core gameplay value and mechanical cohesion. Reduced scope to focus on two fully-realized, polished mechanics—the defensive shield system with UI integration and the damageable environmental area—that naturally complemented each other and created meaningful resource management gameplay loop. This strategic reduction allowed deeper iteration on core systems rather than surface-level implementation of many features. Documented cut features and original design intentions for potential future expansion, treating this as a learning experience in realistic production planning. Allocated saved development time toward playtesting and balance refinement, resulting in two mechanics that felt responsive, well-integrated, and thoroughly tested. This experience taught valuable lessons about feature prioritization, honest progress assessment, and the importance of delivering polished, complete systems over ambitious but unfinished features—lessons that have informed all subsequent project planning and have made me a more realistic and effective designer.

Challenge 4: Shield Depletion Rate Balancing

The Problem: Balancing shield depletion rates to create meaningful resource management without frustrating players required careful tuning of multiple interconnected variables. Initial damage tick rates depleted shields too quickly, making hazardous areas feel punishing rather than strategic. Players abandoned shield pickups entirely, preferring to avoid hazard zones rather than engage with the resource management system. Conversely, when damage rates were reduced too much, shields became overpowered safety nets that removed all tension from traversing dangerous areas. Finding the balance between meaningful consequence and player agency proved elusive through theoretical design alone.

The Solution: Conducted systematic iterative playtesting sessions where I methodically adjusted damage tick rates, tick intervals, shield capacity values, and environmental hazard sizes based on observed player behavior and feedback. Started with instance-editable variables in Blueprint for all balance-related values (damage per tick, tick frequency, shield max capacity, pickup restoration amount), allowing rapid iteration without Blueprint recompilation. Created test scenarios with varying hazard lengths and shield availability to observe how different value combinations affected player decision-making and risk assessment. Implemented comprehensive visual feedback—shield bar color changes indicating depletion severity, audio cues at specific depletion thresholds (75%, 50%, 25%), and particle effects when shields absorbed damage—to clearly communicate resource state. Added subtle screen edge vignette effects when shields were critically low, creating tension without obscuring gameplay. Through testing, discovered optimal balance emerged when hazard traversal consumed approximately 60-70% of full shield capacity, creating meaningful resource cost without guaranteeing failure, encouraging players to seek refills while not making mistakes immediately fatal. This balance maintained player agency while preserving strategic depth, teaching me that effective balancing requires observable player behavior data rather than purely theoretical number adjustment.

What I Learned

This project reinforced the importance of scope management and realistic production planning. My initial ambition exceeded available development time, teaching me to prioritize feature completion over feature quantity. Delivering two fully-functional, polished mechanics proved more valuable than multiple half-finished systems. This lesson has fundamentally shaped how I approach project planning—I now build realistic timelines with buffer for iteration and polish rather than optimistic schedules packed with aspirational features.

Working with Unreal Engine's skeletal mesh system deepened my technical understanding of character rigging, bone hierarchies, and component attachment workflows. Solving the shield attachment challenge required diving into engine documentation, experimenting with different attachment methods (direct bone attachment vs. socket-based attachment), and understanding transform inheritance through skeletal hierarchies. This technical problem-solving experience strengthened my ability to research unfamiliar engine systems independently and troubleshoot complex technical implementation issues—skills that transfer directly to professional development environments.

Implementing interconnected mechanics—where the shield system directly influenced how environmental hazards affected the player—taught me about creating cohesive mechanical loops and emergent gameplay. I learned that truly engaging mechanics emerge from systems that interact meaningfully rather than existing in isolation. The relationship between shield pickups, hazardous areas, and resource management created strategic depth that wouldn't exist with standalone features. Players naturally developed risk-assessment skills evaluating whether current shield capacity justified traversing hazards or if seeking refills first was more optimal.

This project also emphasized the value of clear visual communication and feedback systems. Players needed immediate understanding of shield status, damage sources, and resource availability without overwhelming UI or excessive visual noise. Designing UI elements and feedback systems that communicated complex state information clearly—through color-coded shield bars, audio cues at depletion thresholds, and particle effects during damage absorption—became a critical design consideration that has informed all my subsequent work. I learned that technical implementation means nothing if players can't intuitively understand system states and make informed decisions.
